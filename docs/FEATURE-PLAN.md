# mdPress 特性开发优先级计划

> 更新日期：2026-03-18
> 由产品委员会讨论产出（产品经理、架构师、程序员、测试四个视角）

---

## 背景

v0.2.0 已发布，包含 HTML 输出、GitHub 源、SUMMARY.md 兼容、零配置、serve 预览、quickstart、validate、暗色模式、代码复制、全文搜索、三栏布局、brew 安装等功能。现在需要规划 v0.3.0 及后续版本的特性优先级。

---

## 候选特性评估

### 1. ePub 输出格式

| 维度 | 评估 |
|---|---|
| 用户价值 | **高** — 电子书是多格式出版平台的核心能力，GitBook 用户迁移的刚需 |
| 实现复杂度 | **中** — 架构中已预留 EpubGenerator，OPF/NCX/XHTML 打包有成熟方案 |
| 依赖关系 | 无，可独立开发 |
| 建议版本 | **v0.3.0** |

**产品经理**：README 和迁移文档已承诺 ePub 支持，用户期望值已建立，必须尽快兑现。

**架构师**：v0.2.0 的 `OutputFormat` 接口和 `FormatBuilderRegistry` 已就位，ePub 只需实现一个新的 Builder，架构改动极小。

**程序员**：Go 生态有 `go-epub` 等库可复用，工作量可控。主要工作在 XHTML 转换和 CSS 适配。

**测试**：需覆盖封面、目录、元数据、图片嵌入、多章节结构。推荐用 epubcheck 做合规校验。

---

### 2. 插件系统基础框架

| 维度 | 评估 |
|---|---|
| 用户价值 | **高** — 决定了 mdPress 能否成为可扩展平台，是数学公式、Mermaid 等特性的基座 |
| 实现复杂度 | **高** — 需要设计生命周期钩子、插件发现/加载机制、配置规范 |
| 依赖关系 | 无前置依赖，但它是 KaTeX、Mermaid 等插件的前置 |
| 建议版本 | **v0.3.0** |

**产品经理**：插件系统是从"工具"到"平台"的关键一步。ROADMAP 已列入 v0.3.0，社区期望已建立。

**架构师**：v0.2.0 已在 `internal/plugin/plugin.go` 预留了钩子接口（ConfigLoaded → BeforeParse → AfterParse → BeforeRender → AfterRender → BeforeOutput → AfterOutput）。建议 v0.3.0 先实现本地插件（Go 插件或独立进程），注册中心留到 v1.0.0。

**程序员**：建议先支持"约定目录 + YAML 配置"的插件发现方式，通过 `os/exec` 调用外部进程插件，降低实现复杂度。Go 原生 plugin 包有平台限制（不支持 Windows），需评估替代方案。

**测试**：插件系统需要沙箱测试——恶意插件不能破坏主进程。需要钩子顺序测试、插件冲突测试、错误隔离测试。

---

### 3. Typst 后端

| 维度 | 评估 |
|---|---|
| 用户价值 | **中** — 去除 Chromium 依赖对 CI/CD 和轻量部署很有吸引力 |
| 实现复杂度 | **高** — 需要将整套 CSS 主题体系映射到 Typst 样式系统 |
| 依赖关系 | 无，但受益于插件系统（可作为后端插件） |
| 建议版本 | **v0.4.0** |

**产品经理**：用户最关心的是"能用"和"好用"，Typst 是性能优化和部署体验的提升，但不是功能缺失。优先级低于 ePub 和插件。

**架构师**：ROADMAP 已规划在 v0.4.0，且 `--backend typst` 的设计方向已明确。CSS → Typst 的映射是最大工程量，建议先支持核心主题。

**程序员**：Typst CLI 可通过 `os/exec` 调用，但需要维护一套完整的 `.typ` 模板。预估 3-4 周工作量。

**测试**：需要与 Chromium 后端的输出做视觉回归比对。CJK 排版是重点验证项。

---

### 4. 增量编译

| 维度 | 评估 |
|---|---|
| 用户价值 | **高** — 大型书籍（500+ 页）从 120s 降到 <10s，直接影响写作体验 |
| 实现复杂度 | **中** — 缓存方案已在 ROADMAP 详细设计（SHA-256 哈希 + manifest.json） |
| 依赖关系 | 与 Typst 后端无强依赖，但两者都属于性能优化主题 |
| 建议版本 | **v0.4.0** |

**产品经理**：`serve` 命令已有 `IncrementalBuilder` 雏形，但 `build` 命令还是全量构建。对频繁迭代的作者价值极大。

**架构师**：v0.2.0 的 `IncrementalBuilder` 已在 serve 中实现了文件监听和缓存概念，可以复用到 build 命令。需要设计跨章节依赖（交叉引用、目录）的失效策略。

**程序员**：核心实现约 2 周。难点在于：交叉引用变化可能导致连锁重建、目录页码需全量重算。

**测试**：需要构建正确性回归——增量构建结果必须与全量构建完全一致。需要大型书籍基准测试。

---

### 5. 多页站点模式（类 GitBook SPA 路由）

| 维度 | 评估 |
|---|---|
| 用户价值 | **高** — v0.2.0 已有 `site` 输出，但缺少 SPA 路由和平滑导航 |
| 实现复杂度 | **中** — 需要前端路由和预加载机制，但页面生成框架已存在 |
| 依赖关系 | 与"导航顺滑度优化"(#12) 可合并处理 |
| 建议版本 | **v0.3.0** |

**产品经理**：站点模式是与 GitBook 竞争的关键差异点。用户已反馈导航体验需要改善。

**架构师**：`SiteGenerator` 已实现多页生成，增加 SPA 路由可以通过注入客户端 JS 实现（History API + 预加载）。无需引入前端框架。

**程序员**：核心工作是：页面间平滑切换、预加载相邻页、URL 路由同步。估计 2 周。

**测试**：需要浏览器端 E2E 测试（Playwright/Puppeteer），验证前进/后退、直接 URL 访问、搜索跳转。

---

### 6. VS Code 插件

| 维度 | 评估 |
|---|---|
| 用户价值 | **中** — 提升写作体验，但 CLI 工作流已满足核心需求 |
| 实现复杂度 | **中** — TypeScript 生态，需独立项目维护 |
| 依赖关系 | 依赖 CLI 稳定性，建议 v1.0.0 后考虑 |
| 建议版本 | **v1.0.0** |

**产品经理**：锦上添花。当前用户群以 CLI 高级用户为主，VS Code 插件的 ROI 在用户量更大后才划算。

**架构师**：可以做成轻量包装——调用 mdPress CLI + 预览集成。但需要长期维护，不宜过早引入。

**程序员**：TypeScript + VS Code Extension API，独立仓库管理。工作量约 3 周初版。

**测试**：需要 VS Code 自动化测试框架，不同平台的兼容性测试。

### 7. GitBook/HonKit 迁移工具

| 维度 | 评估 |
|---|---|
| 用户价值 | **高** — 直接降低用户迁移门槛，扩大用户群 |
| 实现复杂度 | **低** — SUMMARY.md 已兼容，主要是 book.json → book.yaml 转换和插件映射提示 |
| 依赖关系 | 无 |
| 建议版本 | **v0.3.0** |

**产品经理**：已有手动迁移文档，但用户希望一键迁移。`mdpress migrate` 命令能大幅降低摩擦。

**架构师**：核心是 `book.json` 解析 + `book.yaml` 生成 + 不兼容插件的警告报告。可复用现有 config 模块。

**程序员**：1 周内可完成。JSON 解析 + 字段映射 + 生成 YAML。

**测试**：收集 10+ 个真实 GitBook 项目做回归测试。需测试各种 book.json 变体。

### 8. 数学公式支持（KaTeX/MathJax）

| 维度 | 评估 |
|---|---|
| 用户价值 | **高** — 技术书籍、学术写作的刚需 |
| 实现复杂度 | **中** — HTML 端注入 KaTeX JS/CSS 即可，PDF 端需 Chromium 渲染 |
| 依赖关系 | 建议依赖插件系统，作为首个官方插件发布 |
| 建议版本 | **v0.3.0** |

**产品经理**：技术和学术用户的高频需求。作为插件系统的首个示范案例，一举两得。

**架构师**：可作为 `BeforeParse` 钩子插件：识别 `$...$` 和 `$$...$$` 语法，注入 KaTeX 渲染。HTML/Site/ePub 通过 CSS+JS 渲染，PDF 由 Chromium 自然支持。

**程序员**：如果走插件路径，需要先完成插件框架。如果独立实现，直接在 Markdown 后处理阶段注入即可，1 周。

**测试**：需要覆盖行内公式、块级公式、复杂公式（矩阵、分式、希腊字母）、各输出格式的渲染一致性。

---

### 9. PlantUML/Mermaid 图表渲染

| 维度 | 评估 |
|---|---|
| 用户价值 | **高** — 技术文档标配，Mermaid 在 v0.2.0 已部分支持（HTML 端 CDN 渲染） |
| 实现复杂度 | **中** — Mermaid 客户端渲染已有，需补 PDF/ePub 的服务端渲染；PlantUML 需外部服务或 JAR |
| 依赖关系 | 建议依赖插件系统 |
| 建议版本 | **v0.3.0**（Mermaid 增强）/ **v0.4.0**（PlantUML） |

**产品经理**：Mermaid 用户基数大且已部分支持，优先完善。PlantUML 需要 Java 依赖，可延后。

**架构师**：Mermaid 在 PDF 中需要预渲染为 SVG（通过 mermaid-cli 或 Puppeteer）。建议作为 `AfterParse` 钩子插件。PlantUML 依赖外部服务，复杂度更高。

**程序员**：Mermaid 服务端渲染可通过 `npx @mermaid-js/mermaid-cli` 实现，但引入了 Node.js 依赖。需评估是否可选。

**测试**：图表渲染需要视觉回归测试。不同 Mermaid 图表类型（流程图、时序图、甘特图）都需覆盖。

---

### 10. 国际化（CLI 输出多语言）

| 维度 | 评估 |
|---|---|
| 用户价值 | **低** — CLI 用户通常能接受英文输出，国际化不影响核心功能 |
| 实现复杂度 | **低** — Go 的 `embed` + 简单 i18n 库即可 |
| 依赖关系 | 无 |
| 建议版本 | **v1.0.0** |

**产品经理**：用户价值有限。项目已有中英双语文档，CLI 输出的国际化优先级低。

**架构师**：`internal/i18n` 模块已存在，扩展到 CLI 输出技术上可行，但涉及全量字符串提取。

**程序员**：工作量不大（约 1 周），但维护成本持续——每次新增功能都要更新翻译文件。

**测试**：需要多语言输出的完整回归测试套件。

---

### 11. PDF 排版质量提升

| 维度 | 评估 |
|---|---|
| 用户价值 | **中** — 页眉页脚 v0.1.0 已支持，水印和自定义页边距是进阶需求 |
| 实现复杂度 | **低** — 基于现有 Chromium 后端的 CSS `@page` 规则扩展 |
| 依赖关系 | 无 |
| 建议版本 | **v0.4.0** |

**产品经理**：企业用户有水印和品牌定制需求。ROADMAP 已规划在 v0.4.0。

**架构师**：水印通过 CSS `::after` 伪元素或 HTML 叠加层实现。自定义页边距通过 `@page { margin: ... }` 实现。改动集中在主题系统。

**程序员**：1 周内可完成。主要是 CSS 模板扩展和 book.yaml 配置项增加。

**测试**：需要 PDF 视觉回归测试，验证不同页面尺寸下的排版效果。

---

### 12. 导航顺滑度优化

| 维度 | 评估 |
|---|---|
| 用户价值 | **高** — 用户已反馈的体验问题，直接影响产品口碑 |
| 实现复杂度 | **低** — 前端 CSS/JS 优化，不涉及后端架构变更 |
| 依赖关系 | 可与多页站点模式 (#5) 合并 |
| 建议版本 | **v0.3.0** |

**产品经理**：用户反馈优先处理。这是留住现有用户的关键。

**架构师**：与 #5 多页站点 SPA 路由一起做，事半功倍。

**程序员**：预加载、过渡动画、滚动位置恢复，纯前端工作，1 周。

**测试**：需要性能基准测试（FCP、TTI）和用户体验测试。

### 13. 自动生成索引/术语表

| 维度 | 评估 |
|---|---|
| 用户价值 | **低** — v0.1.0 已有 GLOSSARY.md 支持，自动索引是出版社级需求 |
| 实现复杂度 | **中** — 需要关键词提取、频率统计、页码映射 |
| 依赖关系 | 无 |
| 建议版本 | **未来** |

**产品经理**：专业出版场景需求，当前用户群较少涉及。

**架构师**：需要在 PDF 输出中实现页码映射，技术上需要两遍渲染。

**程序员**：关键词提取可用 NLP 库辅助，但精度和多语言支持是挑战。

**测试**：需要大量真实书籍验证索引质量。

---

### 14. 多格式同时输出

| 维度 | 评估 |
|---|---|
| 用户价值 | **中** — 方便但非必需，用户可以多次执行 build |
| 实现复杂度 | **低** — `--format pdf,html,epub` 的解析和并行调用 |
| 依赖关系 | 依赖各输出格式的完善（ePub 需先实现） |
| 建议版本 | **v0.3.0** |

**产品经理**：README 示例已展示 `--format pdf,html,site,epub`，用户预期已建立。

**架构师**：`FormatBuilderRegistry` 已支持多格式注册，只需在 build 命令中遍历调用。可并行执行提升速度。

**程序员**：核心改动在 `cmd/build.go`，解析逗号分隔的格式字符串，循环调用对应 Builder。半天工作量。

**测试**：需要验证各格式组合的并行安全性和输出正确性。

---

### 15. GitHub Actions 模板

| 维度 | 评估 |
|---|---|
| 用户价值 | **高** — CI/CD 自动构建是技术团队的标准工作流 |
| 实现复杂度 | **低** — YAML 模板文件 + 文档，无代码改动 |
| 依赖关系 | 无 |
| 建议版本 | **v0.3.0** |

**产品经理**：极低成本、高回报。v0.2.0 ROADMAP 已提及 CI/CD 模板，应尽快提供。

**架构师**：提供 `.github/workflows/mdpress.yml` 模板，涵盖 build + deploy to Pages 流程。

**程序员**：半天工作量。主要是编写和测试 YAML 模板。

**测试**：需要在真实 GitHub 仓库验证 Actions 运行。测试矩阵覆盖 ubuntu/macos。

---

### 16. 在线 Playground

| 维度 | 评估 |
|---|---|
| 用户价值 | **高** — 零门槛体验，对推广和获客价值极大 |
| 实现复杂度 | **高** — 需要 WebAssembly 编译或云端服务、在线编辑器、实时预览 |
| 依赖关系 | 建议在 v1.0.0 稳定后再启动 |
| 建议版本 | **未来** |

**产品经理**：长期来看价值极高，但当前阶段应聚焦核心功能完善。

**架构师**：两条路径——(1) Go → WASM 编译在浏览器运行，但 Chromium PDF 渲染不可能；(2) 云端服务器运行 mdPress，前端展示结果。两者都是独立项目级别的工程。

**程序员**：预估 2-3 个月以上。建议作为独立项目启动。

**测试**：需要安全性测试（用户输入的 Markdown 可能包含恶意内容）、并发测试、资源隔离测试。

## 决策原则

1. **用户承诺优先**：README 和文档中已承诺的功能（ePub、多格式输出）必须优先兑现
2. **平台化基座先行**：插件系统是扩展能力的基石，优先于具体插件功能
3. **用户反馈驱动**：导航优化等已反馈的体验问题优先修复
4. **低成本高回报**：迁移工具、GitHub Actions 模板等低复杂度高价值的特性尽早交付
5. **性能优化延后**：Typst、增量编译等性能优化在功能完善后再集中处理
6. **稳定性兜底**：v1.0.0 聚焦在稳定性、测试覆盖和文档完善，不引入大型新特性
