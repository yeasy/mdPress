## 分析方法论

每项特性从四个角色视角评估利弊，并综合以下维度给出优先级评分（1-10分，10=最紧急）：

- **用户价值**：解决多少用户的多大痛点
- **市场差异化**：相对竞品的独特性和竞争力
- **技术积累**：对未来架构演进的贡献
- **实现成本**：工程量、维护负担、风险
- **时机因素**：市场窗口期、用户承诺兑现

## v0.3 计划特性分析

### 1. ePub 输出

**利**：
- README 已承诺 ePub 支持，用户期望已建立，不兑现将损害信誉
- 电子书是多格式出版平台的核心能力，直接对标 HonKit/Pandoc/Quarto 的全格式矩阵
- Go 生态有成熟库（go-shiori/go-epub），技术调研确认实施可行且风险可控
- 完善 mdPress "一个命令，四种格式"的核心差异化定位
- ePub 3.3 已是 W3C 推荐标准，生态成熟

**弊**：
- 跨平台兼容性测试负担重（Apple Books、Kindle、Kobo 行为差异大）
- 中文字体嵌入和 PKG-026 媒体类型问题是常见拒收原因
- ePub 中 JavaScript 在多数阅读器不支持，交互功能受限
- 长期维护需跟进 ePub 规范演进

**依赖关系**：无前置依赖，可独立开发

**竞品参考**：HonKit 通过 Calibre 支持 ePub（质量一般），Pandoc 原生支持（质量好但配置复杂），Quarto 原生支持。mdBook 仅通过第三方插件 mdbook-epub 支持。

**调研支撑**：技术调研推荐 go-shiori/go-epub + EPUBCheck 5.0.1 验证流程，实施难度评估为"低"。中文字体策略明确（Noto Serif CJK SC + font/sfnt 声明）。

**优先级评分：9/10** — 已承诺功能，架构已预留接口，实施成本低，兑现用户期望的最高优先级。

---

### 2. 插件系统框架

**利**：
- 是从"工具"到"平台"的关键跨越，决定 mdPress 的长期生命力
- KaTeX、Mermaid、PlantUML 等所有扩展功能的技术基座
- 技术调研已完成方案选型（stdin/stdout JSON 协议，类 mdBook 模式），风险可控
- 语言无关的插件协议可吸引多语言社区贡献
- v0.2.0 已在 `internal/plugin/plugin.go` 预留了完整的钩子接口

**弊**：
- 设计复杂度高，JSON 协议需保证向后兼容，一旦发布难以修改
- 子进程模式有启动开销，大型书籍中多插件场景可能影响性能
- 需要编写 Go/Python/Node.js SDK 降低插件开发门槛，增加维护面
- 安全性考量——插件可访问文件系统，需要明确安全边界文档

**依赖关系**：无前置依赖，但它是 KaTeX (#4)、Mermaid (#5)、PlantUML (#12) 的前置条件

**竞品参考**：mdBook 的 preprocessor/renderer 子进程模式已验证可行（40+ 插件生态）；Typst 的 WASM 插件是更先进的方向（1,150+ 包）；Obsidian 的事件系统社区活跃度极高。技术调研还指出 WASM 插件（Extism/Wazero）可作为 Phase 2 演进方向。

**调研支撑**：技术调研推荐 Phase 1 用 stdin/stdout JSON + Go interface 内置插件，Phase 2 引入 WASM 沙箱。Wazero（零依赖 Go WASM 运行时）已达 v1.7.0 可用。

**优先级评分：9/10** — 平台化基座，阻塞多个后续特性，ROADMAP 已承诺，必须在 v0.3.0 交付。

---

### 3. 多页站点 SPA 模式

**利**：
- v0.2.0 的 site 输出缺少 SPA 路由和平滑导航，用户已反馈体验需改善
- 直接对标 VitePress 的亚秒级 HMR 和 GitBook 的流畅导航体验
- 无需引入前端框架，通过 History API + 预加载即可实现
- 与导航顺滑度优化合并处理，事半功倍

**弊**：
- 需要浏览器端 E2E 测试（Playwright/Puppeteer），CI 复杂度增加
- 预加载策略不当可能导致流量浪费和性能问题
- SPA 模式下的 SEO 需要额外处理（预渲染/SSR）

**依赖关系**：与导航顺滑度优化可合并

**竞品参考**：VitePress 混合渲染（首屏静态 HTML + 后续 SPA 导航）是行业标杆；Docusaurus 也是 SPA 架构。GitBook 的导航体验流畅但依赖 SaaS 基础设施。

**调研支撑**：VitePress 方案（按需编译 + 预取视口内链接）值得借鉴。SiteGenerator 已实现多页生成基础。

**优先级评分：8/10** — 用户反馈驱动，竞争力关键，实施成本中等。

---

### 4. KaTeX 数学公式

**利**：
- 技术书籍和学术写作的绝对刚需，缺失直接流失目标用户群
- 作为插件系统的首个官方示范插件，一举两得地验证插件架构
- HTML/Site/ePub 通过 CSS+JS 渲染，PDF 由 Chromium 自然支持，多格式兼容简单
- 竞品中几乎全部支持（Pandoc、Quarto、Typst、mdBook 通过插件），属于"必须有"

**弊**：
- 依赖插件系统先完成，存在排期耦合风险
- KaTeX 与 MathJax 选型需权衡（KaTeX 更快但覆盖面不如 MathJax）
- ePub 阅读器中 MathML 支持不一致（Kindle 不支持 JavaScript）
- 复杂公式在不同输出格式间的渲染一致性验证工作量大

**依赖关系**：依赖插件系统 (#2)

**竞品参考**：Quarto 和 Pandoc 对数学公式的支持是行业标杆（交叉引用、公式编号）；mdBook 通过 mdbook-typst-math 等插件支持；Typst 的数学模式是原生内建。

**调研支撑**：竞品分析将 KaTeX 列为高优先级学习清单第一项。

**优先级评分：8/10** — 刚需功能，但被插件系统阻塞。如果插件系统延期，建议独立实现基础版本。

---

### 5. Mermaid 图表增强

**利**：
- v0.2.0 已有 HTML 端 CDN 渲染基础，增强工作有延续性
- 技术文档标配能力，Mermaid 在 GitHub/Notion/Obsidian 中已广泛普及
- PDF/ePub 的服务端 SVG 预渲染补全多格式一致性
- 与 KaTeX 一起作为官方插件发布，丰富插件生态

**弊**：
- PDF 中需 mermaid-cli 或 Puppeteer 做服务端渲染，引入 Node.js 依赖
- Mermaid 版本更新频繁，CDN 和本地版本需同步维护
- 复杂图表（甘特图、类图）的 SVG 在不同输出格式中的尺寸适配需要调试

**依赖关系**：建议依赖插件系统 (#2)，但可独立实现

**竞品参考**：mdBook 通过 mdbook-mermaid 插件支持；Quarto 原生支持 Mermaid；GitBook 原生支持。

**调研支撑**：竞品分析将 Mermaid 列为高优先级。技术上 mermaid-cli 可通过 npx 调用。

**优先级评分：7/10** — 重要但 v0.2.0 已有基础，增强性质而非从零实现。

---

### 6. GitBook 迁移工具

**利**：
- GitBook 激进涨价（Premium $65/月）导致用户大规模流失，迁移窗口期明确
- 实现复杂度极低（book.json → book.yaml 转换 + 不兼容插件警告），1 周可完成
- `mdpress migrate` 一键迁移可作为获客利器，配合迁移文档和视频教程
- SUMMARY.md 已原生兼容，迁移的核心障碍已消除
- HonKit 用户群也是直接迁移目标

**弊**：
- GitBook 插件生态庞大，不可能 100% 兼容，用户预期管理是挑战
- 迁移工具的长期维护价值递减（一次性使用）
- 需要收集足够多的真实 GitBook 项目做回归测试

**依赖关系**：无

**竞品参考**：无竞品提供类似的迁移工具，这是差异化机会。

**调研支撑**：竞品分析明确指出 GitBook 涨价用户是理想迁移目标群体。已有手动迁移文档（MIGRATION_FROM_GITBOOK.md）。

**优先级评分：8/10** — 极低成本、高回报，市场窗口期明确，应尽早交付。

---

### 7. 多格式同时输出

**利**：
- README 示例已展示 `--format pdf,html,site,epub`，用户预期已建立
- 实现极其简单——解析逗号分隔格式字符串，循环调用 Builder，半天工作量
- `FormatBuilderRegistry` 已支持多格式注册，架构改动最小
- 可并行执行提升构建速度，体现工具的专业性

**弊**：
- 并行构建时资源竞争需处理（Chromium 并发、内存占用）
- 部分格式组合可能产生意外交互（如同时输出 site 和 html 的路径冲突）
- 用户可以多次执行 build 替代，非必需功能

**依赖关系**：依赖 ePub (#1) 先完成才能实现完整的四格式同时输出

**竞品参考**：Quarto 支持多格式输出；Pandoc 通过多次调用或 Makefile 实现。

**调研支撑**：特性计划确认实现复杂度极低。

**优先级评分：6/10** — 成本极低但价值也有限，跟随 ePub 完成后顺手实现即可。

---

### 8. GitHub Actions 模板

**利**：
- 极低成本（YAML 模板 + 文档，无代码改动），半天工作量
- Docs-as-Code 已是行业标准，CI/CD 集成是技术团队的标配工作流
- 大幅降低采用摩擦，新用户可 5 分钟内跑通完整流水线
- 可作为推广素材在 GitHub Marketplace 展示

**弊**：
- 需要在真实 GitHub 仓库验证 Actions 运行
- 需覆盖多种场景（build + deploy to Pages、多格式输出、Chromium 安装）
- 长期需跟进 GitHub Actions 平台变化

**依赖关系**：无

**竞品参考**：Docusaurus、VitePress、mdBook 都提供 GitHub Pages 部署指南。Quarto 有官方 CI 模板。

**调研支撑**：竞品分析将 GitHub Actions 模板列为短期策略重点。行业趋势确认 Docs-as-Code 是标准。

**优先级评分：7/10** — 成本几乎为零、回报确定，应尽早提供。

## v0.4 计划特性分析

### 9. Typst 后端

**利**：
- 消除 Chromium 依赖，Docker 镜像从 +400MB 降到 +50MB，CI/CD 友好
- Typst 编译速度快 5-10 倍（100 页 PDF：Chromium ~30s vs Typst ~5s）
- Typst 支持 PDF/UA-1 无障碍标准和 PDF/A 合规，满足欧盟/美国法规需求
- 两后端长期共存为用户提供灵活选择
- Typst 45k stars，社区快速增长，长期技术投资价值高

**弊**：
- CSS → Typst 样式映射是最大工程量（3 个核心主题，2 周以上）
- Typst 仍处于 0.x 阶段，可能有破坏性变更
- 自定义 CSS 无法直接迁移到 Typst，需要提供等价写法文档
- 需要维护两套渲染管线，长期维护成本翻倍
- 排版结果与 Chromium 不完全一致，可能引发用户困惑

**依赖关系**：无强依赖，但受益于插件系统（可作为后端插件）

**竞品参考**：Typst 定位替代 LaTeX，已有 1,150+ 社区包。Quarto 已将 Typst 作为 PDF 后端选项。Pandoc 3.1.2+ 原生支持 Typst writer。

**调研支撑**：技术调研推荐 CLI subprocess 方式，使用 Pandoc Typst Writer 做 Markdown → Typst 转换。go-typst 包可复用。CJK 排版质量评估 8/10。

**优先级评分：7/10** — 战略价值高但不紧急，v0.4.0 时机合适。

---

### 10. 增量编译

**利**：
- 大型书籍（500+ 页）从 120s 降到 <10s，直接影响写作体验的核心指标
- `serve` 已有 IncrementalBuilder 雏形可复用，实现有基础
- 技术调研提供了清晰的三阶段实施路径（文件哈希 → 内容缓存 → 分布式缓存）
- Turbopack、Astro 等行业标杆证明增量编译是构建工具的核心竞争力

**弊**：
- 交叉引用变化可能导致连锁重建，依赖图追踪复杂
- PDF 页码全局重算是难点（改一章可能影响后续所有章节页码）
- 缓存一致性验证——增量构建结果必须与全量构建完全一致
- 缓存磁盘空间管理需要策略

**依赖关系**：与构建缓存 (#13) 共享基础设施

**竞品参考**：Hugo 的部分渲染策略（只重渲当前页 + 首页 + 最近 10 页）；VitePress 按需编译；Typst 增量编译 <1s。

**调研支撑**：技术调研详细设计了缓存目录结构、依赖图分析和失效策略。推荐阶段一（文件哈希追踪）可实现 3-5x 加速。

**优先级评分：7/10** — 性能关键但当前用户群规模下非首要瓶颈，v0.4.0 合适。

---

### 11. PDF 排版提升

**利**：
- 企业用户的品牌化需求（水印、自定义页眉页脚 logo、页边距）
- 基于现有 Chromium 后端的 CSS `@page` 规则扩展，改动集中且可控
- 实现成本低（1-1.5 周），ROI 高
- 从"能用"到"商用"的关键跨越

**弊**：
- CSS `@page` 在不同浏览器版本间行为可能不一致
- 水印的旋转角度和位置在不同页面尺寸下需精确调试
- 视觉回归测试矩阵较大（3主题 × 5页面尺寸 × 有/无水印 × 有/无分栏 = 60 组合）

**依赖关系**：无

**竞品参考**：Typst 的 PDF 排版质量是行业标杆；Quarto 支持丰富的页面配置；Pandoc 通过 LaTeX 实现高质量排版。

**调研支撑**：长期路线图已设计详细的 book.yaml 配置方案（watermark、margin、columns、header/footer template）。

**优先级评分：5/10** — 重要但不紧急，企业用户群尚小，v0.4.0 时机合适。

---

### 12. PlantUML 支持

**利**：
- 企业级技术文档中使用广泛（UML 类图、时序图）
- 与 Mermaid 互补而非替代，覆盖更复杂的 UML 图
- 通过插件系统实现，验证插件生态的扩展能力

**弊**：
- 需要 Java 依赖（本地 JAR 模式）或网络依赖（在线渲染模式），破坏零依赖定位
- 在线渲染有隐私风险且速度不稳定
- PlantUML 语法比 Mermaid 更复杂，用户群相对较小
- Go 生态中不存在成熟的 PlantUML 实现

**依赖关系**：依赖插件系统 (#2)

**竞品参考**：Quarto 和 Sphinx 支持 PlantUML；mdBook 有第三方插件。Mermaid 在新项目中的采用率已超越 PlantUML。

**调研支撑**：长期路线图推荐优先在线渲染、本地 JAR 可选。但引入 Java 依赖与 mdPress 轻量化定位矛盾。

**优先级评分：4/10** — 用户群较窄，依赖外部服务/运行时，可延后到社区需求明确时再实现。

---

### 13. 构建缓存

**利**：
- 与增量编译互补——增量编译加速单项目，构建缓存加速跨会话（CI/CD 场景）
- CI/CD 中可缓存 `.mdpress-cache/` 目录，大幅减少重复构建
- 与增量编译共享基础设施，额外工作量仅约 1 周
- 缓存策略（SHA256 + manifest）设计已在路线图中明确

**弊**：
- 缓存一致性验证是持续的测试负担
- mdpress 版本升级导致全量缓存失效，用户可能困惑
- 缓存磁盘空间需要配置化管理
- 与增量编译的职责边界需明确

**依赖关系**：与增量编译 (#10) 共享基础设施，建议同版本开发

**竞品参考**：Turbopack 文件系统缓存（Next.js 16.1 稳定）；Farm 1.0 模块级磁盘持久缓存；Vite 内容哈希策略。

**调研支撑**：技术调研设计了三阶段缓存策略。长期路线图定义了完整的失效机制。

**优先级评分：5/10** — 增量编译的自然延伸，但独立价值有限，跟随增量编译一起实现。

## v1.0 计划特性分析

### 14. VS Code 插件

**利**：
- VS Code 是 Markdown 写作者的主要编辑器，集成体验提升用户留存
- 侧边栏实时预览、book.yaml 补全、一键构建等功能降低使用门槛
- 技术调研已明确轻量方案（WebView + CLI 调用，不做 LSP）
- 独立仓库管理，不增加核心代码复杂度

**弊**：
- TypeScript 项目，需要独立的技术栈和维护团队
- 依赖 CLI 稳定性，过早开发会因 CLI 变更导致大量返工
- Webview UI Toolkit 已弃用（2025年1月），需替代方案
- 三平台兼容性测试负担重

**依赖关系**：依赖 CLI 接口稳定（建议 v1.0.0 API 冻结后）

**竞品参考**：Quarto 有完整的 VS Code 集成（代码执行 + 并排预览）；mdBook 无官方 VS Code 插件。Foam 和 Dendron 是笔记类插件的参考。

**调研支撑**：技术调研推荐 CustomTextEditorProvider + WebView 方案，短期不实现 LSP。工作量约 4 周。

**优先级评分：4/10** — 锦上添花，当前 CLI 用户群不强烈需要，等 v1.0.0 稳定后再投入。

---

### 15. 插件注册中心

**利**：
- 从"能扩展"到"好发现、好安装"的关键基础设施
- GitHub-based 方案零运维成本（命名约定 + 索引仓库 + GitHub Actions 抓取）
- `mdpress plugin install/list/remove` 命令提升用户体验
- 为长期生态建设奠定基础

**弊**：
- v0.3.0 插件生态刚起步，注册中心可能"货架空空"
- GitHub-based 方案搜索体验差、元数据分散
- 安全性审核机制缺失（任何人可发布"恶意"插件）
- 过早建设可能投入产出不匹配

**依赖关系**：依赖插件系统 (#2) 已稳定运行一段时间

**竞品参考**：Typst Universe（1,150+ 包，Web 界面 + CLI 安装）；npm registry 模式；Hugo Modules 基于 Go Modules。

**调研支撑**：长期路线图建议 v1.0.0 先做 GitHub-based 方案，插件超过 50 个时考虑自建。

**优先级评分：3/10** — 生态建设需要时间，v1.0.0 初期插件数量不足以支撑注册中心的价值。

---

### 16. CLI 国际化

**利**：
- 中文和日文用户社区的体验提升
- Go 的 `embed` + 简单 i18n 方案，实现成本低（约 1 周）
- `internal/i18n` 模块已存在，扩展到 CLI 技术可行

**弊**：
- CLI 用户通常能接受英文输出，实际用户价值有限
- 每次新增功能都需同步更新翻译文件，持续维护成本
- 非核心功能，分散开发资源

**依赖关系**：无

**竞品参考**：几乎无竞品做 CLI 国际化（Docusaurus/VitePress/mdBook 等均为英文 CLI）。

**调研支撑**：特性计划和长期路线图均将其评为最低优先级 v1.0.0 特性。

**优先级评分：2/10** — 价值有限，可在 v1.0.0 末期按需实现或由社区贡献。

### 17. API 冻结

**利**：
- v1.0.0 的核心承诺，从"尝鲜工具"到"生产级基础设施"的关键转变
- 用户可放心在生产流水线中使用 mdPress，不担心升级破坏
- 遵循 SemVer 建立社区信任，是开源项目成熟度的标志
- 为插件生态提供稳定的接口契约

**弊**：
- 冻结过早可能锁死设计缺陷，需要充分的社区反馈期
- 需要对所有公开接口编写契约测试，工作量约 2 周
- 冻结后新增功能只能通过新增字段/命令，不能修改已有接口
- book.yaml schema 冻结需要提前充分考虑扩展性

**依赖关系**：依赖 v0.3.0 和 v0.4.0 的所有特性已稳定

**竞品参考**：mdBook 的 book.toml 格式长期稳定；Docusaurus 遵循 SemVer 但仍有偶尔的 breaking changes。

**调研支撑**：长期路线图详细定义了需要冻结的接口清单和兼容性承诺。

**优先级评分：8/10** — v1.0.0 的核心意义所在，但时机上必须等其他功能稳定后才能执行。

## 长期特性分析

### 18. 在线 Playground (WASM)

**利**：
- 获客漏斗顶端——零安装体验，30 秒内感受 mdPress 能力
- Pandoc 3.9 和 Typst 已实现 WASM 在线编辑，证明可行性
- Go 1.24 的 `go:wasmexport` 改进 WASM 导出，时机日益成熟
- 对推广获客价值极大，特别是非技术用户

**弊**：
- Go WASM 二进制体积大（2.88-3.1 MiB 起步），移动端体验差
- PDF 输出不可能在 WASM 中实现（Chromium 无法运行），只能预览 HTML
- 工程量巨大（2-3 个月+），需要独立项目级投入
- 安全性挑战（用户输入的 Markdown 可能包含恶意内容）

**依赖关系**：建议在 v1.0.0 稳定后启动

**竞品参考**：Typst Web（app.typst.app）是 Rust + WASM + Leptos 实现的完整编辑器；Pandoc 3.9 的 pandoc.org/app。

**调研支撑**：技术调研推荐分阶段实施——MVP 用 JS + markdown-wasm（31KB），正式版用 Go + TinyGo。Dagger 已成功从 React 迁移到 Go WASM。

**优先级评分：5/10** — 长期价值高但投入巨大，当前阶段应聚焦核心功能。建议 v1.1.0 启动 MVP。

---

### 19. AI 集成

**利**：
- 2025-2026 年 AI 已成为文档工具标配（GitBook Agent、Mintlify AI、Docusaurus AI Search）
- 内容摘要生成对 SEO 和目录页有实际价值
- 通过插件系统集成，不污染核心代码
- 支持本地模型（Ollama）满足隐私需求

**弊**：
- AI 输出不确定性导致测试困难
- API Key 管理和成本控制增加用户使用复杂度
- AI 功能在开源工具中的定位不明确（是否与开源精神冲突？）
- 非核心能力，分散开发资源

**依赖关系**：依赖插件系统 (#2)

**竞品参考**：GitBook AI Agent 主动建议文档更新；Mintlify 月服务 100 万+ AI 查询；Microsoft 365 Copilot Document Writing Agent（2026年4月开放）。

**调研支撑**：竞品分析指出 AI 辅助写作和 AI 搜索将成为文档工具基础能力。长期路线图建议作为可选插件。

**优先级评分：3/10** — 锦上添花，当前优先级远低于核心功能完善。建议 v1.3.0 以插件形式提供。

---

### 20. 协作编辑

**利**：
- 对标 GitBook 和 Google Docs 的实时协作体验
- Git 工作流近期无需 mdPress 改动（纯文档和最佳实践）
- 章节锁模式是务实的中间方案

**弊**：
- 实时协作（OT/CRDT）是独立产品级别的工程（6-12 个月）
- 远超当前团队规模和资源
- 与 mdPress "CLI 工具"的核心定位不符
- 即使实现，也难以与 Google Docs/Notion 的协作体验竞争

**依赖关系**：依赖 Playground/云端服务作为基础

**竞品参考**：GitBook 有实时协作但存在同步 Bug；Typst 的 typst.app 提供在线协作。

**调研支撑**：长期路线图建议近期（Git 工作流）0 工作量，中期（章节锁）4-6 周，远期（CRDT）6-12 个月。

**优先级评分：1/10** — 投入产出比极低，远超当前项目阶段，建议 v2.0+ 再考虑。

---

### 21. 商业化/SaaS

**利**：
- 项目可持续发展的经济基础
- SaaS 版本可基于 Playground 扩展（更大文件、更多格式、自定义域名）
- 企业版（SSO/SAML/审计日志）满足合规需求
- GitBook 的高定价（$65-249/月）证明市场空间存在

**弊**：
- 需要多租户隔离、支付集成、用量计量等全新基础设施
- 商业化与开源社区的平衡需要谨慎处理
- 维护 SaaS 服务的运维负担远超开源工具
- 当前用户基数不足以支撑商业化

**依赖关系**：依赖 v1.0.0 稳定 + Playground + 足够的用户基数

**竞品参考**：Typst 开源编译器 + 付费 Web 应用（typst.app）；GitBook 纯 SaaS；Leanpub 平台模式。

**调研支撑**：长期路线图建议 v1.0.0 后先尝试 SaaS 模式验证市场。

**优先级评分：2/10** — 远期愿景，当前阶段应全力聚焦产品完善和用户增长。

---

### 22. Obsidian/Notion 集成

**利**：
- Obsidian 用户基数巨大，知识管理 → 出版是自然延伸
- Obsidian 已是本地 Markdown，主要处理 Wikilink `[[]]` 语法转换，实现简单
- Notion → mdPress 扩大内容源，降低迁移成本
- 通过 Source 接口扩展实现，架构已有预留

**弊**：
- Notion API 调用和格式转换复杂度高
- Obsidian 社区可能更倾向使用原生发布方案
- 维护多个 Source 实现的长期成本
- Notion API 频繁变化，集成可能不稳定

**依赖关系**：无强依赖

**竞品参考**：无直接竞品提供 Obsidian → 书籍出版的完整工具链，这是差异化机会。

**调研支撑**：长期路线图建议 Obsidian 优先（实现简单、用户需求大），Notion 其次。每个集成约 2-3 周。

**优先级评分：4/10** — Obsidian 集成值得在 v1.1.0 考虑（成本低、用户群大），Notion 可延后。

## 版本分配调整建议

### v0.3.0 调整（2026-08）

**保持不变**：ePub、插件系统、SPA 站点、KaTeX、Mermaid、迁移工具、多格式同时输出

**建议调整**：
- GitHub Actions 模板**提前到 v0.3.0 首批交付**（2026-04），因为成本几乎为零且能立即降低采用摩擦
- 确定 v0.3.0 的交付顺序：GitHub Actions → ePub → 插件系统 → KaTeX/Mermaid → SPA → 迁移工具 → 多格式

### v0.4.0 调整（2026-11）

**保持不变**：Typst 后端、增量编译、PDF 排版提升、构建缓存

**建议调整**：
- PlantUML **延后到 v1.0.0 或更晚**。原因：用户群窄、Java/网络依赖与轻量化定位矛盾、Mermaid 已覆盖大部分图表需求
- 腾出的工期用于 Typst 后端的充分测试和 CJK 排版打磨

### v1.0.0 调整（2027-Q1）

**保持不变**：API 冻结、VS Code 插件、CLI 国际化

**建议调整**：
- 插件注册中心**延后到 v1.1.0**。原因：v1.0.0 时插件生态刚起步，注册中心的"货架"太空，不如等插件数量到 20+ 再建设
- 将 PlantUML 作为可选插件在 v1.0.0 实现（此时插件系统已经过两个版本迭代，足够稳定）
- v1.0.0 的核心精力放在测试覆盖率 ≥90%、API 冻结、文档完善

### 长期调整

**建议调整**：
- 在线 Playground **提前到 v1.1.0**，先出 MVP 版本（JS + markdown-wasm，31KB，1-2 周即可上线），完整版再考虑 Go WASM
- Obsidian 集成 **v1.1.0** 就可启动（实现简单、2-3 周），Notion 集成延后

## 团队角色分歧与共识

### 分歧一：插件系统 vs 独立实现 KaTeX/Mermaid

| 角色 | 立场 |
|------|------|
| PM | 如果插件系统延期，KaTeX/Mermaid 应独立实现（不能让依赖阻塞刚需功能） |
| ARCH | 坚持插件系统先行，KaTeX/Mermaid 作为首批插件验证架构（倒逼插件系统质量） |
| DEV | 独立实现只要 1 周，重构为插件只需半天，建议双轨并行 |
| QA | 独立实现和插件实现是两套测试代码，倾向一步到位走插件路径 |

**最终共识**：插件系统是 v0.3.0 第一优先级。如果插件系统在 2026-05 前未达到可用状态，KaTeX/Mermaid 临时独立实现，后续迁移为插件。

---

### 分歧二：PlantUML 在 v0.4.0 还是延后

| 角色 | 立场 |
|------|------|
| PM | 延后。用户调研中 PlantUML 需求远低于 Mermaid，ROI 不划算 |
| ARCH | 延后。Java 依赖与零依赖定位矛盾，在线渲染有隐私风险 |
| DEV | 中立。实现不难（1.5 周），但维护外部服务依赖的长期成本高 |
| QA | 延后。需要 mock HTTP 服务用于 CI 测试，增加测试基础设施复杂度 |

**最终共识**：PlantUML 延后到 v1.0.0 作为可选插件。如果社区有强烈需求，可在 v0.4.0 末期追加。

---

### 分歧三：插件注册中心的时机

| 角色 | 立场 |
|------|------|
| PM | 延后到 v1.1.0。v1.0.0 时插件太少，"空货架"伤害产品印象 |
| ARCH | v1.0.0 就需要。GitHub-based 方案成本低（2 周），为社区贡献提供基础设施 |
| DEV | 中立。GitHub-based 确实实现简单，但优先级应让位于测试覆盖率 |
| QA | 延后。v1.0.0 应聚焦稳定性，注册中心引入新的测试面（安装/卸载/升级） |

**最终共识**：v1.0.0 实现基础的 `mdpress plugin install` 命令（直接从 GitHub URL 安装），但 GitHub-based 索引仓库和搜索功能延后到 v1.1.0。

---

### 分歧四：Typst 后端的投入时机

| 角色 | 立场 |
|------|------|
| PM | v0.4.0 合适。CI/CD 用户是核心目标群体，Chromium 依赖是明确痛点 |
| ARCH | v0.4.0 合适，但担心 Typst 0.x 的破坏性变更风险 |
| DEV | 建议推迟到 v0.5.0。CSS → Typst 映射工程量可能超预期（5 周） |
| QA | v0.4.0 可以。视觉回归测试框架正好在 v0.4.0 建立 |

**最终共识**：v0.4.0 实现 Typst 后端，但初版只支持 3 个核心主题（technical/elegant/minimal）。自定义 CSS → Typst 的完整支持延后到 v1.0.0。版本锁定 Typst 0.14.2，观察 Typst 1.0 路线图再决定升级策略。

---

### 分歧五：在线 Playground 的优先级

| 角色 | 立场 |
|------|------|
| PM | 尽早启动 MVP。获客价值极大，markdown-wasm 方案成本可控 |
| ARCH | v1.1.0 合适。WASM 方案的技术债不能在核心稳定前引入 |
| DEV | v1.1.0。JS + markdown-wasm 的 MVP 可以在 1-2 周内完成 |
| QA | 延后。Playground 的安全性测试（XSS 防护、资源隔离）需要专项投入 |

**最终共识**：v1.1.0 启动 Playground MVP（JS + markdown-wasm + Monaco Editor），仅支持 HTML 预览。完整功能版（PDF 输出、Go WASM）视 MVP 反馈再决定。

---

### 全局共识

1. **v0.3.0 是最关键版本**——决定 mdPress 能否从工具进化为平台，所有 P0 特性不可延期
2. **轻量化定位不妥协**——任何引入重型外部依赖（Java/大型 Node 模块）的特性降优先级
3. **用户承诺必须兑现**——README 和文档中已承诺的功能（ePub、多格式输出）优先
4. **测试覆盖不是装饰**——每个版本预留 15% 工期用于测试，v1.0.0 之前所有高风险技术债清零
5. **社区反馈驱动长期规划**——v1.x+ 的特性优先级由 GitHub Issues 投票和用户反馈决定
