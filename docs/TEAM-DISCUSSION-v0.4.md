# mdPress 产品委员会讨论纪要 — v0.4 路线图

> 会议日期：2026-03-19
> 参与角色：产品经理（PM）、架构师（ARCH）、程序员（DEV）、测试（QA）
> 当前版本：v0.3.0 已发布
> 核心议题：PDF 中文字体缺失问题 + v0.4 路线图规划

---

## 背景说明

v0.3.0 发布后，用户最集中的反馈是：**PDF 输出中文字体不显示**。根因是 `chromedp.PrintToPDF` 在 headless 模式下无法正确嵌入 CJK 字体，导致生成的 PDF 在没有预装对应字体的环境（如 CI/CD 容器、Windows 默认安装、无中文字体的 Linux）上出现字符缺失、乱码或方块字。

这个问题直接影响 mdPress 的核心卖点（多格式输出 + CJK 友好），必须在 v0.3.1 中修复，同时需要决定 v0.4.0 的中长期 PDF 架构方向。

---

## 议题一：PDF 中文字体问题怎么解决？

**背景**：现有方案 `chromedp.PrintToPDF` 在 headless 无头模式下，系统字体无法被 Chromium 找到并嵌入 PDF，导致 CJK 文本显示为空白或乱码。候选方案：

- **方案 A**：继续 chromedp，修复字体嵌入问题
- **方案 B**：切换到 WeasyPrint（Python，CSS Paged Media）
- **方案 C**：切换到 Typst（Rust，新一代排版引擎）
- **方案 D**：切换到 wkhtmltopdf
- **方案 E**：多后端架构（默认 chromedp，可选 Typst）

---

### 产品经理（PM）

这个问题我认为分两层来看：**短期修复**和**长期架构**。

短期来说，v0.3.1 必须给用户一个能用的方案，不能让 README 上写着"中文友好"但实际生成的 PDF 是一堆方块。用户在 GitHub Issues 里已经开始投票了，口碑损失每天都在累积。

长期来说，我不倾向于 B（WeasyPrint）和 D（wkhtmltopdf）：WeasyPrint 引入 Python 运行时依赖，破坏了我们"单一二进制"的核心卖点；wkhtmltopdf 基于 Qt WebKit，已经实质性停止维护（最后一个正式版是 2020 年），排版质量也远不如 Chromium，是在走历史的回头路。

我支持**方案 E（多后端）**，但执行顺序要分阶段：v0.3.1 先把 chromedp 的字体嵌入问题修好（方案 A 的修复），给现有用户一个快速出口；v0.4.0 把 Typst 后端做出来，作为"无 Chromium 依赖"的替代方案。这样既能快速响应用户痛点，又不仓促地替换掉已经稳定的 Chromium 渲染管线。

竞品视角：Quarto 已经将 Typst 作为 PDF 后端选项，Pandoc 3.1.2+ 原生支持 Typst writer。我们不能在这个趋势上落后太多。

**PM 结论**：v0.3.1 修 chromedp 字体问题（方案 A 的增强版），v0.4.0 引入 Typst 后端（方案 C），最终形成方案 E 的多后端格局。

---

### 架构师（ARCH）

PM 说的方向大体对，但我需要补充技术细节，因为"修复 chromedp 字体嵌入"这件事本身有几个层次，难度差异很大。

**chromedp 字体问题的根因分析**：

Chromium headless 模式查找字体走的是操作系统的字体目录（`/usr/share/fonts`、`~/.fonts` 等），在标准 Docker 镜像、GitHub Actions runner、很多 CI 环境里，根本没有安装 Noto CJK 或类似的中文字体包。`PrintToPDF` 不会自动嵌入字体，它只是把 Chromium 渲染的内容转成 PDF，而渲染时找不到字体就显示空白。

**修复路径有三种**：

1. **字体回退方案**：在生成 HTML 时，把中文字体（如 Noto Serif CJK SC）通过 `@font-face` + base64 内联到 HTML 里，让 Chromium 从 HTML 内容本身加载字体而不依赖系统。缺点是内联字体会让 HTML 体积暴增（一个中文字体子集约 200-500KB），但 PDF 阶段这只是临时文件，可以接受。

2. **字体文件挂载**：让 mdPress 二进制内嵌（`embed.FS`）一套精简的 CJK 字体子集，在运行时写入临时目录，并通过 `--user-data-dir` 或 chromedp 的字体路径参数告知 Chromium。技术可行，但字体子集化工具链（pyftsubset/font-spider）需要额外维护。

3. **chromedp 启动参数**：通过 `--font-render-hinting=none`、`--disable-font-subpixel-positioning` 等 flag 改善渲染，但这不能解决字体不存在的根本问题。

我的判断：**方案 1（@font-face 内联）是 v0.3.1 的最快路径**，一周内可以交付，不引入新依赖，生成过程中的 HTML 是临时文件所以体积无所谓。

关于 Typst 后端（方案 C）：技术调研已经充分，Typst 的中文支持通过 `(name: "Noto Serif CJK SC", covers: "cjk")` 配置，内置字体发现机制不依赖系统字体，这是对 CI 环境最友好的方案。但 Typst 仍是 0.x 版本，我建议版本锁定在 0.14.2，等 Typst 1.0 路线图更明朗后再考虑升级。

关于方案 B（WeasyPrint）：CSS Paged Media 的排版质量不如 Chromium，而且 Python 依赖违背我们的单一二进制原则，直接排除。

关于方案 D（wkhtmltopdf）：停止维护，排版引擎落后，不考虑。

**ARCH 结论**：v0.3.1 用 `@font-face` 内联解决 chromedp 字体问题；v0.4.0 并行实现 Typst 后端，通过 `--backend=typst` 启用；多后端架构在 `PdfBackend` 接口上抽象，chromedp 和 Typst 均实现该接口。

### 程序员（DEV）

我来估工作量，同时补充一些 ARCH 没提到的实现细节。

**v0.3.1 的 @font-face 内联方案**：

具体做法是在 PDF 渲染的 HTML 模板里注入一段 CSS：

```css
@font-face {
  font-family: 'Noto Serif CJK SC';
  src: url('data:font/woff2;base64,...') format('woff2');
  unicode-range: U+4E00-9FFF, U+3400-4DBF; /* CJK 统一汉字 */
}
```

难点在于**字体子集化**：完整的 Noto Serif CJK SC 有 20MB+，全量内联不现实。我们需要在构建时用 `pyftsubset` 或 `woff2` 工具生成一个包含常用汉字（GB 2312 的 6763 个常用字）的子集，大约 1-2MB，base64 后约 1.4-2.7MB，内联进 HTML 后 Chromium 能正常渲染。

工期预估：**5-7 个工作日**（含字体子集生成脚本、HTML 模板修改、多平台测试）。

这个字体文件可以用 `go:embed` 直接打包进 mdPress 二进制，不需要用户额外安装任何依赖。

**v0.4.0 的 Typst 后端**：

调用链：`Markdown → AST → chromedp HTML → 临时 .typ 文件 → Typst CLI → PDF`

实际上不应该这样做。正确的链路是：`Markdown → Pandoc（markdown-native → typst）→ 注入 mdPress 主题 → Typst CLI → PDF`。

但是这里有个复杂性：我们现有的三套 CSS 主题（technical、elegant、minimal）都是面向 HTML/CSS 的，完全移植到 Typst 需要重写。这是 Typst 后端工期最长的部分，预估 **3-4 周**，不是技术难题而是工作量大。

另外，Typst 二进制本身是 Rust 编译的，我们需要在 CI 里为 linux/amd64、linux/arm64、darwin/amd64、darwin/arm64、windows/amd64 五个平台分别准备 Typst 二进制，并在 mdPress 发布时一起打包。这个分发策略需要提前设计。

**DEV 结论**：支持方案 E，v0.3.1 用内联字体方案，工期约一周；v0.4.0 Typst 后端预估 4-5 周含主题适配和多平台分发，初版只支持三个主题，自定义 CSS 延后。

### 测试（QA）

我重点关注测试策略，因为 PDF 中文字体这个问题之所以到 v0.3.0 才被发现，核心原因是**我们的测试环境有中文字体，用户的不一定有**。这是测试环境与生产环境的偏差，必须在测试体系上修正。

**针对 v0.3.1 的测试要求**：

1. 在 GitHub Actions 的 ubuntu-latest runner 上（默认无中文字体）生成包含中文的 PDF，验证字符可见性。这个测试用例此前根本不存在。

2. 建立 PDF 字体嵌入验证脚本：使用 `pdffonts`（poppler-utils）或 `pdfinfo` 检查 PDF 中的嵌入字体列表，确认 CJK 字体被正确嵌入。这个检查应该加入 CI pipeline。

3. 测试矩阵扩展：
   - 有中文字体的 macOS（当前已有）
   - 有中文字体的 Linux（当前已有）
   - **无中文字体的 Docker Alpine 容器**（新增，最能复现用户场景）
   - **GitHub Actions ubuntu-latest**（新增，常见 CI 场景）

**针对 Typst 后端（v0.4.0）的测试要求**：

1. **视觉回归测试**：需要建立 Chromium 后端和 Typst 后端的输出对比基准。两者不可能像素级一致，但关键内容（标题层级、代码块、表格、中文段落、页眉页脚）必须等价。推荐用 `pdfimages` + 图像哈希比对，允许 5% 的差异阈值。

2. **CJK 专项测试套件**：包含简体中文、繁体中文、日文、韩文的测试文档，验证字符不缺失、排版不错乱、标点不溢出。

3. **跨平台一致性**：同一份文档在 macOS/Linux/Windows 三平台用 Typst 后端生成的 PDF，中文内容不应有差异（Typst 内嵌字体，理论上平台无关）。

**QA 结论**：同意方案 E 的分阶段思路，但强调在 v0.3.1 合并前必须补充无中文字体环境的 CI 测试用例，这是防止同类问题再次发生的底线。

### 议题一共识

**团队一致决定**：

1. **v0.3.1（紧急修复）**：采用 `@font-face` + base64 内联字体子集方案修复 chromedp 字体嵌入问题。字体子集涵盖 GB 2312 常用汉字（~6763 字），用 `go:embed` 打包进二进制，确保零额外依赖。

2. **v0.4.0（战略升级）**：引入 Typst 后端，通过 `pdf.backend: typst`（book.yaml）或 `--backend=typst`（CLI flag）启用。初版支持三个核心主题，Chromium 后端保留作为默认值（成熟稳定，适合复杂 CSS 场景）。

3. **排除方案**：WeasyPrint（Python 依赖）和 wkhtmltopdf（停止维护）不进入考虑范围。

4. **测试强制要求**：v0.3.1 合并条件之一是 CI 新增无中文字体环境的 PDF 字体嵌入验证测试，且必须通过。

## 议题二：后续特性开发顺序

### 产品经理（PM）

梳理当前特性欠账和市场机会，我建议以下优先级顺序：

**P0（必须，用户有强烈痛点）**：
1. PDF 中文字体修复（已在议题一决定）
2. KaTeX 数学公式支持——技术文档用户的刚需，竞品全部支持，缺失直接流失用户
3. Mermaid PDF/ePub 服务端渲染——v0.3.0 的 Mermaid 只在 HTML 里能显示，PDF 里是空白

**P1（重要，有竞争差异）**：
4. Typst PDF 后端——消除 CI/CD 环境的 Chromium 依赖，这是企业用户最关心的
5. 增量编译——大型书籍用户的核心痛点，`serve` 已有雏形

**P2（锦上添花）**：
6. PDF 排版提升（水印、页边距定制）——企业品牌化需求
7. 构建缓存——跟增量编译共享基础设施，顺手做

我注意到当前路线图里 KaTeX 被列为依赖插件系统的特性。我的意见是：插件系统如果稳定了就走插件路径，但不能让插件系统成为 KaTeX 的阻塞项——KaTeX 优先级太高了，可以先独立实现再迁移。

---

### 架构师（ARCH）

从技术依赖关系来说，我给出有依赖约束的顺序：

**第一批（无前置依赖，可立即启动）**：
- PDF 字体修复（v0.3.1）
- Mermaid 服务端 SVG 渲染（在 PDF/ePub 构建时，通过 `mermaid-cli` 预渲染为 SVG，注入 HTML 后再交给 Chromium）
- GitHub Actions 模板更新（把 Chromium 字体安装步骤加进去，顺手修）

**第二批（插件系统就绪后）**：
- KaTeX 作为官方插件发布
- Mermaid 迁移为插件（当前独立实现可先发版，后迁移代价低）

**第三批（独立但工程量大）**：
- Typst 后端——需要主题适配，3-4 周
- 增量编译——`IncrementalBuilder` 雏形可复用，2-3 周

技术上有一个值得关注的点：Mermaid 的 PDF 服务端渲染需要 `mermaid-cli`，后者依赖 Node.js。这和我们的"零外部依赖"定位有冲突。可行的替代方案是：先用 chromedp 的无头浏览器跑一段 JavaScript 来渲染 Mermaid SVG（利用已有的 Chromium 依赖），不引入额外的 Node.js。这个技术路径需要单独设计。

---

### 程序员（DEV）

我从实际可交付的角度补充几点：

**KaTeX 独立实现**的方案其实很简单：在 goldmark 的渲染阶段，识别 `$...$` 和 `$$...$$` 语法（goldmark-mathjax 或自己写 AST transform），在 HTML 模板里注入 KaTeX 的 CSS 和 JS（本地 vendor 或 CDN 可选），PDF 交给 Chromium 自然渲染。整个独立实现约 **3-5 天**。后续迁移为插件时，把这段逻辑包装成 JSON stdin/stdout 接口即可，改动量很小。所以 PM 说的"不让插件系统阻塞 KaTeX"完全可行。

**Mermaid 服务端渲染**：利用 chromedp 本身的 JavaScript 执行能力，在内存中渲染 Mermaid 为 SVG，然后替换掉 HTML 里的 Mermaid 代码块，最后再走 PrintToPDF。这样完全不需要 Node.js，估计 **3-5 天**。

**增量编译**的关键路径是文件哈希追踪：给每个源文件计算 SHA256，存入 `.mdpress-cache/hashes.json`，构建时跳过未变化文件。难点在 PDF——改了第三章的内容，页码可能从第四章开始全部变了，所以 PDF 要么全量重构，要么只支持 HTML/ePub 增量编译。建议第一版只做 HTML 和 Site 的增量，PDF 继续全量，这样可以在 1-2 周内交付有实际价值的版本。

---

### 测试（QA）

我的关注点是测试可维护性和回归风险。

Mermaid 服务端渲染和 KaTeX 是两个新增的重量级渲染路径，它们都需要**视觉对比测试**——不能只靠"跑通了不报错"来验收，必须有人眼可见的参考截图基准和自动比对脚本。

增量编译是风险最高的特性：缓存一致性 bug 会导致用户生成内容不正确但看不出来（文件生成了，但内容是旧的）。必须有专门的**增量 = 全量等价性测试**：同一份书，先全量构建得到结果 A，再修改一个文件后增量构建得到结果 B，验证 B 与从零全量构建的结果 C 完全一致。

**QA 建议的顺序**：优先交付测试成本低的特性（KaTeX 独立实现、Mermaid 利用现有 Chromium），把测试复杂度高的特性（Typst 后端、增量编译）放在有足够测试基础设施之后。

## 议题三：v0.3.1 紧急修复版包含什么？

### 产品经理（PM）

v0.3.1 的定位是**紧急修复版**，不是小版本特性迭代。范围控制是关键——不能趁机塞一堆新东西进去。

**必须包含**（核心修复）：
- PDF CJK 字体嵌入修复（这是主要理由）
- Mermaid 在 PDF 和 ePub 中显示为空白的修复（用户反馈频率第二，且和字体问题一样影响"可用性"）

**应该包含**（低风险、高价值）：
- KaTeX 数学公式基础支持（独立实现，不依赖插件系统，实现干净）
- GitHub Actions CI 模板更新（把字体安装步骤加进去，极低改动量）
- 已知崩溃类 bug 修复（如果有的话）

**不应该包含**：
- Typst 后端（工程量大，v0.4.0 的事）
- 插件系统架构变动（破坏稳定性）
- 任何新的输出格式

发布时间目标：**v0.3.0 发布后 2-3 周内**发布 v0.3.1，让用户感受到我们对反馈的响应速度。

---

### 架构师（ARCH）

同意 PM 的范围界定，从技术角度我来确认可行性和风险。

**PDF 字体修复**：方案明确（`@font-face` 内联），改动集中在 PDF HTML 模板和字体资源打包，不涉及核心渲染逻辑，风险低。

**Mermaid PDF 渲染**：利用 chromedp 执行 JavaScript 渲染 Mermaid 为 SVG 的方案，实现上是在 `PdfBuilder` 的 HTML 生成阶段增加一步 SVG 替换，不改变渲染管线的主流程，风险可控。需要注意并发安全：如果用户配置了并行构建，chromedp 不能并行执行多个 JS 渲染任务，需要加锁或串行化。

**KaTeX**：在 goldmark 管线里增加一个 AST transformer，识别数学语法并注入渲染所需的 HTML 标记，再在模板里引入 KaTeX 的本地 vendor 文件（避免 CDN 依赖）。改动集中，不影响已有输出。

**架构风险点**：KaTeX 的 vendor 文件（JS + CSS + 字体）大约 500KB，加上 CJK 字体子集约 1-2MB，v0.3.1 之后 mdPress 二进制体积会从当前约 15MB 增长到约 17-19MB。这需要在发布说明里明确说明。

### 程序员（DEV）

考虑到这些改动相对独立，可以并行开发：一个人做字体修复，另一个做 KaTeX，在最后两天合并集成测试。如果只有一个人，就按上述顺序串行，优先保证字体修复，KaTeX 如果时间紧可以滑到 v0.3.2。

## 议题四：v0.4.0 聚焦什么主题？

### 产品经理（PM）

在议题一和议题二的讨论基础上，v0.4.0 的主题已经比较清晰了，我来做一个定性的定位。

v0.3.0 完成了"从工具到平台"的基础建设（插件系统、ePub、KaTeX、Mermaid、迁移工具）。v0.3.1 修复了阻塞用户使用的核心问题。

那么 **v0.4.0 应该聚焦"专业化与企业就绪"**：

1. **Typst 后端**：消除 Chromium 依赖，让 mdPress 能在任何 CI/CD 环境里零配置运行，这是从"开发者个人工具"走向"团队工具链"的关键一步。

2. **增量编译**：500 页以上的书籍从 120 秒降到 10 秒以内，这是专业写作用户（技术书籍作者、企业内部文档团队）的核心诉求。

3. **PDF 排版专业化**：水印、自定义页眉页脚、多栏布局，让 PDF 输出达到商用标准，不再是"凑合能用"的状态。

4. **可访问性合规**：欧盟 EAA 已于 2025 年 6 月生效，美国 ADA Title II 截止 2026 年 4 月，PDF/UA 合规是企业客户的硬需求。Typst 后端天然支持 PDF/UA-1。

v0.4.0 的**目标用户画像**：工程团队的技术文档 Lead，需要在 CI/CD 里自动生成 PDF 版 API 手册，对输出质量和法规合规有要求。

---

### 架构师（ARCH）

我完全同意 PM 的定位，从架构角度来说 v0.4.0 还有一件重要的基础工作：**建立视觉回归测试基础设施**。

Typst 后端、PDF 排版提升、增量编译这三件事都涉及 PDF 渲染结果的变化，没有一套可靠的视觉回归框架，我们就是在盲飞。需要建立：

- **PDF 截图基准**：对每个主题、每种代表性内容（代码块、表格、数学公式、中文段落、图表）生成基准截图
- **自动差异检测**：使用 `pdftoppm` 转图片后做像素级比对，允许 2% 以内的差异（字体渲染 hinting 差异）
- **CI 集成**：每次 PR 自动运行视觉回归，差异超出阈值时阻止合并

这个基础设施在 v0.4.0 之前不存在，v0.4.0 引入 Typst 后端是建立它的最佳时机。

另外，v0.4.0 我建议在架构层面做一个清理：当前 `PdfBuilder` 和 `HtmlBuilder` 里有一些重复的 CSS 处理逻辑（主题加载、变量替换），在引入 Typst 后端之前，抽出一个 `ThemeResolver` 接口，让三种 PDF 后端（chromedp、typst、未来可能的 WeasyPrint）共享主题加载逻辑。这不是"为了重构而重构"，是为了 Typst 后端的实现不造成代码重复。

---

### 程序员（DEV）

v0.4.0 的技术挑战我这里列几个需要提前决策的问题：

**Typst 二进制分发**：Typst CLI 是 Rust 编译的，我们需要决定是：(a) 用户自行安装（`brew install typst`），检测不到时给出明确错误信息；(b) mdPress 在首次使用 `--backend=typst` 时自动下载对应平台的 Typst 二进制（类似 `playwright install`）；(c) 把 Typst 二进制打包进 mdPress 发布包（体积会增加约 20MB）。

我倾向于 (a) 先做，最简单，用户可以 `brew install typst` 或 `cargo install typst-cli`。等社区需求明确后再考虑 (b) 的自动下载。

**增量编译的 PDF 问题**：如前述，PDF 页码全局性，增量做不了。第一版增量编译明确只支持 HTML 和 Site，PDF 始终全量，在文档里说清楚。后续 Typst 增量编译（Typst 原生支持 <1s 增量）可以作为 v0.4.0 末期或 v0.5.0 的特性。

**水印实现**：通过 CSS `::after` 伪元素实现，用绝对定位覆盖页面。Chromium 后端已经支持，只需要在 book.yaml 里加 `pdf.watermark.text`、`pdf.watermark.opacity`、`pdf.watermark.angle` 配置项，1 周内可完成。

总工期预估：Typst 后端（4-5 周）+ 增量编译 HTML/Site（2 周）+ PDF 排版提升（1 周）+ 视觉回归基础设施（1 周）= **约 8-9 周**，目标 2026 年 11 月发布。

### 测试（QA）

v0.4.0 是测试负担最重的版本，我需要提前拉响警报：

Typst 后端意味着同一份内容要用两套渲染管线生成 PDF，两套管线的测试矩阵是当前的两倍。在没有自动化视觉回归的情况下，这会严重拖慢发布速度。

**QA 对 v0.4.0 的核心诉求**：

1. **视觉回归测试基础设施必须在 Typst 后端代码合并之前就位**，而不是之后补。ARCH 提到了，我完全支持，这是本版本的技术债务控制底线。

2. **增量编译等价性测试**：构建一个包含 50+ 章节的大型测试书籍，验证"改动第 N 章后增量构建的结果 == 从零全量构建的结果"，并且这个测试要在 CI 中以 `go test -timeout 10m` 的形式运行。

4. **版本回退测试**：v0.4.0 升级后，用 v0.3.x 的 book.yaml 配置文件必须能正常工作（向后兼容）。

## 议题五：是否调整项目定位？

### 产品经理（PM）

这是最重要的战略议题，值得认真讨论。

**当前定位**："单一二进制、零配置的多格式书籍出版工具（PDF/HTML/ePub/Site）"

**需要调整吗？**

我认为定位**方向不变，但表达要升级**：

1. **中文/CJK 定位应该被强化**。我们刚讨论了半天 PDF 中文字体问题，这恰恰说明 CJK 排版是我们在认真解决、而竞品往往忽视的问题。这应该变成一个正面的差异化卖点，而不是遮遮掩掩的技术问题。README 里应该有专门一段说："mdPress 原生支持中文、日文、韩文排版，字体嵌入、标点避头尾、竖排（规划中）均有专项优化。"

2. **"企业就绪"应该被提前到 v0.4.0 就强调**。水印、PDF/UA 合规、CI/CD 友好（Typst 后端）这些特性合在一起，可以支撑"企业版文档工具"的定位叙事。

3. **不要过早强调"平台"**。插件系统还刚起步，说"我们是文档工具平台"有点超前。继续聚焦在"出版工具"这个更具体、更容易让用户理解的定位上。

**我不建议做的调整**：
- 不要向"站点生成工具"偏移（Docusaurus、VitePress 在那里太强了）
- 不要向"笔记工具"偏移（Obsidian、Notion 已经占据那个心智）
- 不要做 SaaS（当前用户规模支撑不了，也会分散精力）

---

### 架构师（ARCH）

从技术架构的角度，项目定位会影响我们的技术选型边界。

当前定位的技术约束是"单一二进制、零外部依赖"，这决定了：
- Chromedp 可以（Chromium 是可选外部依赖，有清晰的安装文档）
- Typst 可以（用户自行安装，类似 Chromium）
- WeasyPrint 不行（Python 运行时不可接受）
- PlantUML 不行（Java 不可接受）
- mermaid-cli 不行（Node.js 不可接受，所以我们走了 chromedp JS 执行路径）

如果定位不变，这些技术边界就不变。我支持这个约束，它帮助我们做出一致的技术决策。

但有一点我想提出来讨论：**Chromium 本身也是一个很重的依赖**。很多用户装了 mdPress 才发现还要装 Chrome，体验断层明显。v0.4.0 的 Typst 后端如果做好了，我们应该考虑把 Typst 作为**推荐默认后端**（对于能安装 Typst 的用户），Chromium 作为复杂 CSS 场景的可选后端，而不是相反。这是一个定位上微妙但重要的信号：我们在走向更轻量、而不是继续绑定 Chromium。

---

### 程序员（DEV）

定位讨论从 DEV 视角来说，我最关心的是：**定位是否影响我们对技术债务的处理方式**。

如果定位向"企业就绪"靠拢，意味着：
- 错误信息要更友好（现在很多 panic 直接暴露给用户，不够专业）
- 配置文件的向后兼容要更严格（book.yaml 的 schema 变动要有迁移路径）
- 日志和 debug 模式要更完善（企业用户排查问题需要）

这些不是新功能，是"产品质量"层面的工作，容易在路线图里被忽视。我建议在 v0.4.0 里预留 20% 工期专门做这类质量提升，而不是把所有时间都花在新特性上。

关于"CJK 优先"的定位：我完全支持，而且 Go 生态在 CJK 方面有优势（goldmark 的 Unicode 处理、全文搜索的 CJK 分词）。把这个变成显性定位，会吸引更多中文技术文档团队，这是一个蓝海市场。

---

### 测试（QA）

我从质量保证的角度谈定位调整的影响：

如果强化"CJK 优先"定位，**我们需要专门的 CJK 测试套件**。目前测试用的样本书籍主要是英文内容，中文测试覆盖率低。这和"CJK 优先"的定位自相矛盾。

具体来说，需要建立：
- 包含简体中文、繁体中文、日文、韩文的标准测试书籍
- 覆盖各种排版场景：长段落、混排英文、标点避头尾、代码注释中文
- 四种输出格式（PDF/HTML/ePub/Site）的 CJK 渲染验证

如果强化"企业就绪"定位，还需要：
- PDF/UA 合规验证的自动化
- 大批量构建的稳定性测试（100 次连续构建不崩溃）
- 内存泄漏检测（长时间运行的 `serve` 命令）

这些测试工作量不小，建议在路线图里显式分配工期，而不是让 QA 在发布前夜突击。

---

### 议题五共识

**项目定位：方向不变，表达升级**

**保持不变的核心定位**：
"单一二进制、零配置、多格式（PDF/HTML/ePub/Site）书籍出版工具"

**强化的差异化定位**：
- **CJK 优先**：原生支持中文、日文、韩文排版，字体嵌入有专项保障，是市场蓝海

**明确不做的方向**：
- 不做 SaaS 或云端服务（当前阶段）
- 不做站点生成工具（与 VitePress/Docusaurus 直接竞争不现实）
- 不做笔记或知识管理工具

**落地行动**：
- v0.3.1 发布时同步更新 README：增加"CJK 字体支持"专节
- v0.4.0 发布时更新定位描述：突出"CI/CD 友好"和"企业就绪"
- 建立专门的 CJK 测试套件（v0.3.1 开始，QA 主导）
